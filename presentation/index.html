Objectifs du Projet

Automatisation Intelligente

Migrer les règles métier complexes du code applicatif vers le moteur Oracle via des Triggers et Procédures pour garantir l'intégrité absolue des données.

Sécurité Multi-Niveaux

Implémenter un modèle RBAC (Role-Based Access Control) robuste, isolant les données sensibles par profil utilisateur (Étudiant, Prof, Admin).

Optimisation des Flux

Assurer des temps de réponse minimaux via l'indexation et l'usage de RefCursors pour le transfert de données vers l'interface Streamlit.

Discipline Académique

Contrôler automatiquement l'assiduité, les capacités des cours et le respect des prérequis pédagogiques.

Profils de Sécurité

Tables & Rôles : Noyau Identité

USER_ACCOUNT

Stockage des identifiants et hashs de mots de passe. Pilote le routage vers les profils Oracle.

  • PK: user_id
  • Role Check: Student/Prof/Admin

STUDENT / PROF / ADMIN

Tables de profils étendus liées au login. Séparent les données personnelles du compte de sécurité.

  • FK: login_code reference user_account

Tables & Rôles : Structure Académique

DEPARTEMENT / FILIERE

Organisation hiérarchique. Définit l'ancrage administratif des cours et des professeurs.

YEAR / SEMESTRE / SECTION

Temporalité et logistique. Découpent le cursus en unités de temps et groupes d'étude.

COURSE / PREREQUISITE

Le catalogue des modules. Gère les dépendances (Un cours B nécessite le succès au cours A).

Tables & Rôles : Flux Opérationnel

INSCRIPTION_REQUEST / COURSE_RESULT

Moteur transactionnel. Trace les demandes, les acceptations et l'état final (Valid/Failed) de l'étudiant.

SEANCE / ATTENDANCE

Suivi logistique et disciplinaire. Monitoring des absences qui impacte directement le résultat académique via trigger.

UNBLOCK_REQUEST

Piste d'audit administrative pour la réactivation des dossiers étudiants bloqués.

Intelligence Triggers : Intégrité Académique

trg_prof_max_3_courses & trg_check_prof_dept_alignment

Garantit la charge de travail (max 3 modules/an) et que le professeur enseigne uniquement dans son champ d'expertise départemental.

trg_max_7_courses_per_semestre

Empêche la surcharge des filières en limitant le nombre de modules actifs par semestre via une fonction autonome.

Intelligence Triggers : Logique de Flux

trg_check_seance_overlap

Détection de conflits temps-réel : bloque la création d'une séance si la salle ou le professeur est déjà occupé sur ce créneau.

trg_check_prerequisite & trg_block_missing_prerequisite

Vérification automatique des acquis : refuse l'insertion d'une demande d'inscription si les modules dépendants ne sont pas validés.

Intelligence Triggers : Automatisation Disciplinaire

trg_3_consecutive_absences

Compound Trigger : Analyse l'historique et injecte 'FAILED' dans les résultats après 3 absences.

trg_check_course_capacity

Gestion des places : appelle une fonction autonome pour vérifier le quota avant d'accepter un étudiant.

trg_justify_absences_unblock

Séquence de déblocage : régularise automatiquement l'historique d'assiduité après intervention admin.

trg_accept_inscription_init

Initialisation : crée les lignes de présence prévisionnelles dès l'acceptation de la demande.

Indexation & Performance

Rôle de l'Indexation

  • Optimisation Query : Réduction drastique des temps de scan sur les tables volumineuses comme `ATTENDANCE`.
  • Index Composites : Créés sur les clés étrangères (student_id, seance_id) pour accélérer les jointures des vues.

Sécurité du Schéma

Les index ne servent pas qu'à la vitesse ; ils garantissent l'unicité des relations métier (Contraintes Unique Index) et évitent les verrouillages inutiles lors des transactions concurrentes, assurant la stabilité de la base.

Synonymes Publics

L'Art de l'Abstraction

Le système crée automatiquement des Synonymes Publics pour chaque table et vue.

Sécurité

Masque le propriétaire réel (YAHYA_ADMIN). L'application ignore où se trouve physiquement la table.

Simplicité

Permet aux utilisateurs app_student et app_prof d'écrire SELECT * FROM COURSE au lieu de SELECT * FROM YAHYA_ADMIN.COURSE.

Méthode Sécurité : Utilisateurs Applicatifs

Utilisateurs Oracle Dédiés

app_admin : Le gestionnaire de la plateforme.

app_prof : L'accès pédagogique restreint.

app_student : L'accès consultation minimale.

app_auth : Le portier (ne lit que les logins).

"Principe du moindre privilège : chaque utilisateur ne possède que les droits strictement nécessaires à sa fonction."

Méthode Sécurité : Définition des Rôles

ROLE_ADMIN

  • - CRUD total sur tables.
  • - Select sur toutes les vues.
  • - Execution des procédures admin.

ROLE_PROF

  • - Update Attendance.
  • - Execution sp_submit_grade.
  • - Select sur vues v_prof_*.

ROLE_STUDENT

  • - Insert Request Inscription.
  • - Update Password perso.
  • - Select sur vues v_student_*.

Méthode Sécurité : Flux de Connexion

1. app_auth

Vérifie le mot de passe et récupère le rôle.

2. Switch User

L'app se reconnecte avec le pool correspondant.

3. Session Role

Les droits Oracle sont activés nativement.

Méthode Sécurité : Script security.sql

Le script security.sql automatise le durcissement de la base :

GRANT CREATE SESSION TO app_users;

Autorise la connexion physique.

GRANT ROLE_STUDENT TO app_student;

Liaison de l'utilisateur au profil.

CREATE PUBLIC SYNONYM FOR tables;

Génération de la couche d'abstraction.

REVOKE ALL FROM app_users;

Nettoyage des droits par défaut.
